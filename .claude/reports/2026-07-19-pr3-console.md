## Архитектура (locked E1-E4)
- **E1** новый app (не webapp/Telegram). **E2** HMAC-подпись только серверная
  (`BANK_HMAC_SECRET` в server-env, роуты `/api/bank/*` подписывают и форвардят).
  **E3** operator-гейт (`CONSOLE_OPERATOR_PASSWORD`) — серверный, на КАЖДОМ
  bank-запросе (`bankRoute.operatorOk`), не обходим с клиента. **CORS** (замечание
  #1) — same-origin proxy для публичных чтений.
- **Инвариант #2** (sign-forward идентичные байты): `bankForward` сериализует
  тело ОДИН раз в Buffer, подписывает и форвардит те же байты.

## Как тестировал (всё зелёное локально)
```
typecheck ✓ · lint ✓ · build ✓ (12 роутов) · vitest 8 passed
```
- **Signing byte-for-byte** (`bankSign.test`): подпись == golden-вектор из
  Python `bank/deps.compute_signature` + byte-sensitivity (не тавтология).
- **Invariant #2** (`bankForward.test`): recompute подписи над РЕАЛЬНО
  форварднутыми байтами == отправленная подпись; тело = payload JSON один раз;
  503 без секрета (fetch не вызван).
- **Operator-гейт** (`operator.test`): точный пароль / неверный / пустой-null /
  сервер без пароля.
- **Секрет не в клиентском бандле**: `grep .next/static` по `BANK_HMAC_SECRET` /
  `CONSOLE_OPERATOR_PASSWORD` — пусто (серверные чанки — ожидаемо держат).
- **coverage_ratio=null** (supply=0) — UI показывает «no supply», без деления на 0.
- **set-state-in-effect** починен по причине (setState в `.then`-колбэках), НЕ
  даунгрейдом правила.
- **UUID-guard** на `/api/bank/withdrawals/[id]` — id валидируется до вставки в
  upstream-путь (защита от path-манипуляции).
