# P1-A — Киперы: heartbeat, UZS, резерв

Дата: 2026-07-23 · Инженер · Гейт-1 (план) → после «go» Капитана: код → Гейт-2 (диф)
Редакция 2 — после BLOCKED ревью: внесены БЛОКЕРЫ 1–3 и МИНОРЫ 1–5.

## Цель

Ни один из трёх канонических фидов никогда не публиковал значение, а существующий
кипер XAU молчит, когда значение не менялось (`keeper-xau.ts:98–102`), — при окнах
протухания `maxAgeXau=3600` / `maxAgeReserve=3600` это гарантированно убивает
Exchange каждые выходные и блокирует минт постоянно. Цель: все три фида живы
непрерывно — первая публикация + heartbeat (принудительная публикация при подходе
возраста ончейн-записи к `maxAge`, независимо от дельты), с гардами по возрасту
источника и скачку значения.

Фиды умирают от ДВУХ дыр, спека лечит обе: (а) дыра дельты — ранний выход при
неизменном значении; (б) дыра источника — гард `MAX_SOURCE_AGE=3600`
(`keeper-xau.ts:76–77`) роняет кипер до heartbeat-решения в любой тихий период:
heartbeat Chainlink XAU/USD = 22 ч 20 мин 16 с (отклонение 0.3%,
data.chain.link), легитимный возраст записи доходит до ~22 ч (максимум — на
выходных, когда отклонения нет и работает только heartbeat). Последняя цена
закрытого рынка — корректная цена: ретранслируем, а не отказываемся.

## Решения Капитана, на которых стоит спека

- **К-7:** один ключ-постер на все три аттестора; роль деплоера не отзываем
  (ручной запасной путь). Env: `KEY_XAU` / `KEY_UZS` / `KEY_RESERVE` — три имени,
  одно значение (гибкость разделения сохраняется без правки кода).
- **К-8:** `MAINNET_RPC_URL` — keyed-провайдер (Alchemy/Infura, бесплатный тариф),
  безключевой публичный RPC — запасной в конфиге.
- **К-1:** банк подписывает выписку раз в сутки; кипер резерва повторяет то же
  число каждые 30 минут. Источник числа — `RESERVE_GRAMS` в `contracts/.env`
  (подтверждено ревью) + гард скачка по БЛОКЕРУ 3.
- **К-3:** резерв 5 кг = `RESERVE_GRAMS=5000`; демо-выпуск ~половина ёмкости (P1-C).

## Скоуп PR

**Входит:**

1. **Heartbeat в `keeper-xau.ts`** — правка логики пропуска:
   - читать ончейн `updatedAt` (сейчас читается только `answer`,
     `keeper-xau.ts:93`);
   - публиковать, если `delta > minDelta` **ИЛИ** `onchainAge >= HEARTBEAT_AGE`;
   - проверка возраста строго ДО раннего выхода по дельте;
   - `HEARTBEAT_AGE` — env, дефолт `1800` (подтверждён ревью);
   - гард первой публикации (`current == 0`) не меняется.
2. **БЛОКЕР 1 — дыра источника (XAU):** `MAX_SOURCE_AGE` дефолт `3600` →
   **`259200` (72 ч)**. Обоснование по первоисточнику (data.chain.link):
   отклонение 0.3%, heartbeat 22 ч 20 мин 16 с — heartbeat срабатывает и при
   закрытом рынке, публикуя ту же цену, поэтому штатный максимум возраста
   записи ≈ 22 ч (не 49 ч заморозки: заморожен бывает рынок, не фид). 72 ч =
   запас >3× heartbeat источника; **нижняя граница дефолта — 26 ч независимо от
   того, что покажет замер**. Градация лога по возрасту: ≤ 23 ч — штатно,
   обычная строка (золото просто не сдвинулось на 0.3%; такое бывает и в разгар
   вторника); (23 ч, 72 ч) → ретранслируем с пометкой `source heartbeat
   exceeded, relaying last known (age Ns)`; > 72 ч → отказ (`source broken`).
   Порог пометки стоит выше heartbeat источника, а не выше часа: на эти логи
   опирается алертинг P4-A (МИНОР 4), ложный сигнал в будни недопустим.
3. **БЛОКЕР 2 — один ключ, гонка за nonce:** новый вход `scripts/keeper-all.ts`,
   прогоняющий XAU → резерв → UZS **последовательно, одним процессом** (один
   nonce-поток — гонка структурно невозможна). Каждый фид в своём try/catch:
   отказ одного не отменяет остальные. Одиночные входы (`keeper:xau` и т.д.)
   сохраняются для ручных прогонов; расписанию (P4-A) отдаётся **только**
   `keeper:all` — требованием, записанным здесь.
4. **МИНОР 3 — часы:** `now` для ончейн-возраста = timestamp последнего блока
   zkSync (не системные часы); для возраста Chainlink-записи = timestamp
   последнего блока мейннета; системные часы — только для разбора календарной
   `Date` ЦБ (там минуты не играют).
5. **МИНОР 4 — контракт кодов возврата** (для алертинга P4-A): одиночный кипер —
   `0` = опубликовал, `2` = сознательный пропуск (unchanged / heartbeat не дозрел /
   wild deviation), `1` = отказ или ошибка (источник сломан, env, RPC, tx).
   `keeper:all` — per-feed строка результата в лог + код выхода = худший по фидам
   (`1` если хоть один отказ, иначе `0`/`2`).
6. **`scripts/keeper-uzs.ts`** — новый кипер: официальный JSON ЦБ РУз
   (`cbu.uz/ru/arkhiv-kursov-valyut/json/USD/`, без ключа; поля проверены живым
   запросом 23.07: `Rate` = сум за 1 USD строкой, `Date` = `DD.MM.YYYY` даты
   курса) → `UzsUsdFeed` (`0x6373…a24aD`, dec 8, значение = `Rate` × 1e8).
   Гарды: положительное, формат ответа, **МИНОР 1: возраст курса ЦБ — щедрый
   дефолт `604800` (7 суток)**: гард ловит «API сломан», а не «сегодня выходной»
   (замер ревью: за сб/вс API отдаёт курс с исходным `Date` последнего рабочего
   дня; в понедельник возраст = 3 суток). Свежесть ончейн-записи держит
   heartbeat (`HEARTBEAT_AGE_UZS=86400`), «курс от пятницы» показывается в
   Console (P2-A), а не блокирует публикацию. Гард скачка — как у XAU.
7. **`scripts/keeper-reserve.ts`** — новый кипер: `RESERVE_GRAMS` из env →
   `ReserveAttestor` (`0x9413…7d1B`, dec 0). Гарды: целое положительное;
   heartbeat `1800`. **БЛОКЕР 3 — гард скачка обязателен:** при `current > 0`
   изменение > `MAX_RESERVE_JUMP_BPS` (дефолт 1000 = 10%) отвергается, если
   `RESERVE_CONFIRM` не равен точному новому значению; опечатка в env не уедет
   в цепь автоматически, смена резерва — сознательная операция. Первая
   публикация (`current == 0`) — без гарда (это сид, P1-C).
8. **Замер Chainlink XAU/USD (К-8) — фоновое подтверждение, не ворота:**
   дефолт зафиксирован из документированных параметров источника (п. 2), Фаза 1
   замером не блокируется. Наблюдение в фоне: снятие `updatedAt` мейннет-фида
   через keyed RPC каждые ~6 ч пт→пн; результат — максимум наблюдаемого разрыва
   + будничная каденция числами в отчёте P1-A. Ожидание по параметрам: максимум
   ≈ 22 ч; если наблюдение покажет больше 26 ч — дефолт пересматривается вверх
   отдельным решением. Снимает LIKELY-статус поведения Chainlink на выходных.
9. **Общий код киперов** — только реально общее (heartbeat-решение, гарды,
   коды возврата): один модуль `scripts/keeper-lib.ts`, три тонких скрипта +
   `keeper-all.ts` поверх. Никакой «платформы киперов».
10. **Тесты** (см. тест-план) + npm-скрипты `keeper:uzs`, `keeper:reserve`,
    `keeper:all`.
11. **Доки:** шапки скриптов, `contracts/.env.example` — новые ключи (имена и
    комментарии, без значений): `KEY_UZS`, `KEY_RESERVE`, `RESERVE_GRAMS`,
    `RESERVE_CONFIRM`, `HEARTBEAT_AGE*`, `UZS_ATTESTOR_ADDRESS`,
    `RESERVE_ATTESTOR_ADDRESS`, `MAX_RESERVE_JUMP_BPS`.

**Явно НЕ входит:**

- Расписание/сервисизация на сервере (cron, мониторинг, долив ETH) — P4-A.
  Требования к P4-A, записанные здесь как контракт: расписывать только
  `keeper:all`; **МИНОР 2: интервал прогона ≤ (maxAge − HEARTBEAT_AGE) / 3** =
  (3600 − 1800)/3 = **600 с (10 мин)**. Проверка стоит чтения, транзакции —
  только публикации: прогон раз в 10 мин не меняет число транзакций (~48/сутки
  на XAU и резерв, та же смета 0.02 ETH/30 дней), но переживает два сорванных
  прогона подряд. Без этого правила запас по heartbeat нулевой.
- Выдача `POSTER_ROLE` ключу-постеру и его финансирование — P1-B (ops).
  **МИНОР 5: платит ключ-постер, не деплоер** — задача для P1-B сформулирована
  точно: перевести постеру **≥ 0.03 ETH** (0.02 смета + запас), контроль
  остатка — на постере (P4-A).
- Первая аттестация резерва, казна Exchange, демо-клиенты — P1-C.
- Изменения контрактов (`*.sol`) — не требуются.
- Две строки в Console («выписка от…» / «опубликовано…») — Фаза 3.

**Один PR.** Причина изменения одна: «фиды обязаны жить непрерывно». Линия
разреза (heartbeat XAU | два новых кипера) — в кармане, если потребуется.

## Критерии приёмки

PR готов, когда:

1. Тест «дельта в пределах ε, но ончейн-запись старше `HEARTBEAT_AGE` →
   публиковать» **падает на текущем коде** (вывод в отчёте) и зелёный после
   правки — доказательство фикса дыры дельты.
2. Тест «Chainlink-запись возрастом 50 ч (выходные) при `MAX_SOURCE_AGE=72 ч` →
   ретранслируем, а не отказ» зелёный — доказательство фикса дыры источника.
3. Весь новый/изменённый код покрыт тестами (см. тест-план), `npm test` и
   линт/тайпчек в `contracts` зелёные, вывод приложен.
4. Поведение существующих гардов не изменилось: источник старше лимита → отказ;
   скачок > порога → отказ; первая публикация → пост; свежая неизменная запись →
   пропуск.
5. Ручной прогон на Sepolia (после P1-B): `keeper:all` публикует первые
   значения трёх фидов последовательно (nonce-гонки нет по построению);
   повторный прогон при неизменном значении и дозревшем heartbeat публикует
   снова (`roundId+1`), свежий прогон — честный пропуск с кодом `2`.
6. Замер Chainlink приложен числами (фоновое наблюдение п. 8 скоупа); дефолт
   `MAX_SOURCE_AGE` зафиксирован из документированных параметров с полом 26 ч.
7. Гард резерва проверен вручную один раз на Sepolia: `RESERVE_GRAMS` с +50%
   без `RESERVE_CONFIRM` → отказ; с `RESERVE_CONFIRM` → пост.

## Риски

- Поведение Chainlink на выходных — из первоисточника (heartbeat 22 ч 20 мин
  срабатывает и при закрытом рынке): штатный максимум разрыва ≈ 22 ч, дефолт
  72 ч покрывает его с запасом >3×; фоновое наблюдение п. 8 подтверждает, при
  разрыве > 26 ч дефолт пересматривается вверх отдельным решением.
- Нонс-гонка — LIKELY (не воспроизводилась): закрыта структурно (`keeper:all`),
  воспроизводить не требуется.
- Смета: ~97 публикаций/сутки × 0.0000069 ETH ≈ 0.02 ETH/30 дней с баланса
  **постера** (≥ 0.03 ETH — задача P1-B).
